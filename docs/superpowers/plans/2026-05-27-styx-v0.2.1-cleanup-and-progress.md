## Phase B — Budget redesign (Tasks B1–B5)

### Task B1: Message-count tracker schema

Replace the token-based fiction with reality. CLI wrappers (claude / codex / agy) don't report token counts, so styx's `EstTokensIn/Out = len(s)/4` is garbage in, garbage out. What styx CAN count accurately: API calls (one per `tracker.Record`).

**Files:**
- Modify: `internal/budget/budget.go`
- Modify: `internal/budget/budget_test.go`

**Schema change:** the `usage` sqlite table already records one row per call. We don't need to change the table — just change what `State()` returns. Add a `MessageCount` field. Keep token columns for backward-compat (they're meaningless but harmless).

**Window change:** introduce two windows per channel — `WindowSession = 5h` and `WindowWeek = 168h`. `State()` returns both counts.

**Cap change:** new struct fields `MessagesPer5h` and `MessagesPerWeek`. Reads from `routing.toml` `[budget.<channel>]` table.

```go
type State struct {
    Channel        string
    SessionCount   int           // messages in last 5h
    SessionLimit   int           // configured cap
    WeeklyCount    int           // messages in last 168h
    WeeklyLimit    int           // configured cap
    SessionPct     float64       // SessionCount/SessionLimit * 100
    WeeklyPct      float64       // WeeklyCount/WeeklyLimit * 100
    LimitHit       bool          // either pct >= 100
    CooldownUntil  time.Time
}
```

**API:**

```go
package progress

import (
    "fmt"
    "io"
    "os"
    "sync"
    "sync/atomic"
    "time"
)

// Tracker is the top-level progress emitter. One per styx invocation.
type Tracker struct {
    w        io.Writer    // typically os.Stderr
    isTTY    bool
    verbose  bool         // --verbose: show prompt sizes, model parameters
    quiet    bool         // --quiet: suppress all progress (only final results print)
    mu       sync.Mutex
    active   *Stage
}

// Stage represents an ongoing operation with a spinner (if TTY).
type Stage struct {
    parent  *Tracker
    name    string
    started time.Time
    done    atomic.Bool
    stop    chan struct{}
    info    []string         // extra detail lines printed under the spinner
}

func New(w io.Writer, quiet, verbose bool) *Tracker
func Default() *Tracker  // returns os.Stderr Tracker with quiet/verbose from env

// Stage opens a new step. If a stage is already active, this closes the parent
// implicitly (we don't model nesting in v1).
func (t *Tracker) Stage(name string) *Stage

// Info adds a sub-detail line under the current spinner. Visible in TTY mode
// when --verbose; always visible in non-TTY mode.
func (s *Stage) Info(format string, args ...any)

// Done completes the stage with a summary.
func (s *Stage) Done(format string, args ...any)

// Fail completes the stage as failed.
func (s *Stage) Fail(err error)

// Quiet returns a no-op Tracker (for tests + --quiet mode).
func Quiet() *Tracker
```

**Behavior:**

TTY mode (stderr is a terminal, not `--quiet`):
```
[styx] Drafting initial response... ⠋ 1m23s
       prompt: 1247 tokens
       model: agy:default
```
Spinner advances every 100ms. Time updates every second.

Non-TTY mode (pipe / redirect / `--quiet`):
```
[styx] Drafting initial response... (started)
[styx] Drafting initial response... (1m23s, complete: 8432 tokens)
```
No animation. Start + end lines only.

`--verbose` flag adds Info() detail lines in both modes.

### Task C3: Wire progress into channel adapters

**Files:**
- Modify: each `internal/channel/<x>/<x>.go`

The channels themselves shouldn't know about progress — that couples the abstraction too tightly. Instead: wrap channel Send calls at the caller (research.Loop, plan, build, auto adapters) with a progress.Stage.

Actually — the cleanest approach is a `channel.Channel` decorator:

```go
// In internal/channel/decorator.go
type WithProgress struct {
    Inner    Channel
    Tracker  *progress.Tracker
    Label    string
}

func (w *WithProgress) Send(ctx context.Context, req Request) (Response, error) {
    s := w.Tracker.Stage(w.Label + " calling...")
    if req.Prompt != "" {
        s.Info("prompt: ~%d tokens", len(req.Prompt)/4)
    }
    resp, err := w.Inner.Send(ctx, req)
    if err != nil {
        s.Fail(err)
        return resp, err
    }
    s.Done("returned ~%d tokens", len(resp.Text)/4)
    return resp, nil
}
```

Then `cmd/styx/dispatch.go` `defaultChannels()` wraps each channel in `WithProgress`.

### Task C7: execute.Apply progress + stderr stream coordination

**Files:**
- Modify: `internal/execute/execute.go`

`Apply` already streams claude's stderr live. The progress.Stage for "running claude" should NOT compete with that stream. Solution: progress.Stage prints the start line ("[styx] Applying plan via claude..."), pauses the spinner during the run (since claude's own stderr is the progress), and prints the done line after.

```go
func Apply(ctx context.Context, o Options, prog *progress.Tracker) (string, error) {
    s := prog.Stage("Applying plan via claude")
    s.Info("plan size: %d bytes", len(o.PlanContent))
    // ... existing logic, claude's stderr streams to os.Stderr below the spinner ...
    s.Pause()  // stop animation while child writes to stderr
    err := cmd.Run()
    s.Resume() // optional; mostly we're going straight to Done
    if err != nil {
        s.Fail(err)
        return "", err
    }
    s.Done("done (%v)", time.Since(s.Started))
    return stdout.String(), nil
}
```

Add `Pause()` / `Resume()` to Stage in progress package.
