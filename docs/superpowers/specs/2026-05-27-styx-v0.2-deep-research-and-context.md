# Styx v0.2 — Deep Research + Saturated Context Handoff

| Field | Value |
|---|---|
| Date | 2026-05-27 |
| Author | Ishaan Batra (brainstormed with Claude) |
| Status | Approved for planning |
| Builds on | v0.1.0 (commit `ef714fe`, tag `v0.1.0`) |
| Scope | Slices 1+2+3 of the post-v0.1 vision: deep research, saturated-context Claude handoff, and full-pipeline autonomy (`styx auto <goal>`) |
| Prerequisite | Antigravity CLI (`agy`) v1.0.2+ installed at `~/.local/bin/agy` (already present) |

---

## 1. Summary

v0.1 shipped a foundation that routes work between Claude, Codex, Gemini, and Ollama via a hand-curated rules table. But the verbs are shallow, manual, and chained by hand. v0.2 deepens each verb AND chains them autonomously:

- **`research`** becomes a convergence loop: agy (Antigravity CLI, Gemini Pro) drafts; codex critiques; agy revises; repeat until codex's structured critique has no BLOCKING/IMPORTANT findings (capped at 6 rounds). `--deep` adds citation chasing: each cited source URL gets fetched and re-summarized into a Sources Appendix.
- **`plan`/`build`** stop handing Claude a vanilla prompt. Instead they materialize a per-project intel index (file tree, module purposes, conventions, key symbols, recent commits, open TODOs) cached in `~/.config/styx/state/intel/<project>/`, render it to `.claude/context.md` for Claude Code to auto-load on startup, and inline the same payload into one-shot `claude -p` calls. The index auto-refreshes when stale (>5 commits or >7 days).
- **`styx auto <goal>`** is new: a full pipeline that chains research → plan → execute → test → review → commit → PR autonomously. Each stage has fix-loops (failed tests trigger Claude to fix; blocking review findings trigger a re-pass). State is persisted per run under `.styx/runs/<run-id>/`, so a killed pipeline resumes with `styx auto --resume <id>`. Individual verbs (`research`, `plan`, `build`, `review`) remain usable standalone for granular control.
- The Gemini channel is **rewritten** to use the `agy` CLI (Antigravity, the gemini-cli successor) — Google is sunsetting gemini-cli on 2026-06-18.

`deep-research` (the v0.1 verb that opened browser tabs) is **removed**. Its useful behavior — multi-source synthesis — is replaced by the citation-chasing path of `research --deep`.

## 2. Goals

1. **Leverage subscriptions at depth, not breadth.** A research call should consume 5-15 LLM invocations across multiple channels, iterating until convergence — not the 2 calls v0.1 makes.
2. **Remove human-in-the-loop entirely on the autonomous path.** `styx auto <goal>` runs research → plan → execute → test → review → commit → PR with no human prompts unless a gate fails. No more browser tabs in research, no manual hand-off between verbs.
3. **Preserve granular control via standalone verbs.** Every step is also a verb the user can run alone: `research`, `plan`, `build`, `review`. `styx auto` is the composition; the parts are still usable.
4. **Saturate Claude's context.** Before Claude runs (plan, build, or any auto stage that uses Claude), styx has already (a) read the repo, (b) summarized modules, (c) extracted conventions, (d) surfaced recent activity, and (e) materialized this into a markdown file Claude Code loads natively.
5. **Resumable pipelines.** A `styx auto` run that gets killed mid-pipeline can be resumed from the last completed stage. State is per-run under `.styx/runs/<run-id>/`.
6. **Stay compatible with v0.1.** All existing verbs still work; the intel index is purely additive; routing rules get migrated automatically.
7. **Migrate off gemini-cli before Google sunsets it.** The new Gemini channel calls `agy` directly. The HTTP-API fallback is removed (it routed to the free dev tier, which doesn't match the "use the subscription" goal).

## 3. Non-goals

- **MCP server for live intel querying.** v0.2 renders the index to markdown; an MCP path stays open for later.
- **Cross-project context.** v0.2 indexes one project at a time. The Hoot trio still has three separate indexes.
- **Semantic search of past briefs.** The index references briefs by filename but no FTS yet.
- **Resumable partial intel builds.** A killed `styx intel` starts over (atomic write protects the prior index). Pipeline runs (`styx auto`) ARE resumable; intel rebuilds within a pipeline are not.
- **Voice / proactive / daemon (slice G).** Out of scope.
- **Goal decomposition.** `styx auto <goal>` runs one pipeline per invocation. Splitting a large goal into multiple coordinated pipelines is out of scope for v0.2.
- **Multi-branch / parallel pipelines.** v0.2 runs one pipeline at a time per project. Concurrent runs on the same project are rejected.

## 4. Decisions log

| # | Decision | Rationale |
|---|---|---|
| 1 | Convergence-based research loop (not fixed rounds) | "Deep" means driven to quality, not driven to a count. Critic returns no blocking + no important → stop. |
| 2 | Hard cap at 6 rounds | Prevents runaway cost on adversarial topics where critic keeps finding nits. |
| 3 | Cached structural index (not live-Pro-every-call) | Plan/build runs frequently; 30-90s of agy latency on every call is unacceptable. Cache + auto-refresh is the right tradeoff. |
| 4 | Index handoff via `.claude/context.md` | Idiomatic to Claude Code; auto-loaded; durable; greppable. |
| 5 | Codex as the critic (not Ollama 14b) | Different training from agy/Gemini → finds disagreements that matter. Ollama 14b stays as a free fallback when codex is capped. |
| 6 | Rename gemini channel to `agy`, drop HTTP-API fallback | Google sunsetting gemini-cli 2026-06-18. Free dev API doesn't match "use the subscription" goal. |
| 7 | `--add-dir` instead of inlining file content for agy | agy has native workspace awareness; we leverage it. |
| 8 | Auto-refresh on >5 commits OR >7 days | Empirical: 5 commits is the threshold where conventions/symbols meaningfully drift; 7 days catches abandoned indexes. |
| 9 | `deep-research` verb removed (becomes `research --deep`) | Browser-in-loop is the exact thing user wants gone. `--deep` flag enables citation chasing autonomously. |
| 10 | Don't clobber user-authored `.claude/context.md` | Write to `.claude/context.styx.md` if a user-authored file exists; reference via `@import` from the canonical one. |
| 11 | `styx auto <goal>` is full-autonomy by default | User chose autonomy as the default endpoint. Granular control comes from running individual verbs, not from `auto` flags. |
| 12 | Pipeline state lives in `.styx/runs/<run-id>/` (project-local) | Per-run dir under the repo, not under `~/.config`. Travels with the project (gitignored), enables resume. |
| 13 | Auto creates a feature branch automatically | `styx/<run-id>-<slug>` branch per run. Avoids dirtying the user's current branch. Final stage opens a PR from this branch. |
| 14 | Fix-loops cap at 5 attempts (test) and 3 attempts (review) | Empirical safety nets. After exhaustion, pipeline halts with state preserved for resume after human inspection. |
| 15 | Non-interactive code execution uses `claude --dangerously-skip-permissions` | Required for headless autonomous Claude runs. Documented as a deliberate trust choice (the user accepts that auto runs are unsupervised). |

## 5. Verb shape changes

### `research <query> [--deep]`

Replaces v0.1's two-call shallow flow.

**Convergence loop:**

```
round 0: agy -p "<research prompt>" --add-dir <project>
         draft_0 = result

loop round = 1..6:
    codex critique of draft_{round-1}:
      system: "Return JSON: {blocking: [string], important: [string], nits: [string]}"
      content: draft_{round-1}
    parse critique_{round}; if JSON fails, fall back to keyword extraction

    if critique_{round}.blocking is empty AND critique_{round}.important is empty:
        CONVERGED. break.

    if content-hash(draft_{round-1}) == content-hash(draft_{round-3}):
        OSCILLATING. mark "convergence_failed" and break.

    agy revises:
      "Revise the draft below. Address every BLOCKING and IMPORTANT finding.
       Keep what works. New draft below.
       --- CRITIQUE ---
       {critique_{round}}
       --- DRAFT ---
       {draft_{round-1}}"
    draft_{round} = result
```

**Output brief structure:**

```markdown
# Research Brief

**Query:** <q>
**Date:** ...
**Status:** converged (after N rounds) | oscillating (drafts cycling) | max_rounds_exhausted (hit cap of 6)
**Channels:** drafter=agy, critic=codex

---

## Final Synthesis
<draft_N>

---

## Convergence Trace
| Round | Blocking | Important | Nits | Verdict |
|---|---|---|---|---|
| 1 | 3 | 5 | 2 | revised |
| 2 | 1 | 4 | 3 | revised |
| 3 | 0 | 0 | 5 | CONVERGED |

## Remaining Nits (non-blocking)
- ...

## Sources (when --deep)
- <url-1>: <summary>
- ...

## Disagreements (when relevant)
If agy and codex independently disagree, capture it explicitly.
```

**`--deep` flag:** after the convergent draft, every URL/citation in the draft gets resolved:

```
for each unique URL in draft:
    fetched_content = curl URL
    summary = agy -p "summarize this source in 200 words, focus on claims relevant to: <q>"
                  --add-dir <(tmpfile containing fetched_content)
    append to Sources section

run one final critic pass on the now-source-rich draft
```

### `plan <description>`

```
1. project = project.Current()
2. if intel.IsStale(project):
       intel.Rebuild(project)         # in-place stderr progress
   index = intel.Load(project)
3. brief = brief.LoadLatest(project)
4. context_md = intel.Render(index, brief, recent_commits, open_todos)
5. write context_md to <project>/.claude/context.styx.md
   if <project>/.claude/context.md exists and isn't ours:
       ensure it @imports context.styx.md (idempotent)
   else:
       write context_md to <project>/.claude/context.md directly
6. prompt = inline context_md + plan instructions + description
7. call claude -p via router (existing v0.1 path), with proj.Path as WorkingDir
8. write result to project's plans dir
```

### `build [target]`

```
1. project = resolveTarget(target)
2. ensure intel fresh (same as plan step 2)
3. write context_md (same as plan step 5)
4. exec claude (interactive) with WorkingDir=project.Path
   Claude Code auto-loads .claude/context.md on startup
```

### `intel <project> [--force]`

```
1. resolve project (alias OR pwd)
2. build:
   a. walk repo (respect .gitignore), file_tree
   b. sniff conventions (deterministic, no LLM)
   c. recent commits via git log
   d. open TODOs via grep
   e. external deps from manifests
   f. parallel agy module summaries (semaphore-limited, e.g., 4 concurrent)
   g. single agy "key symbols" call
3. atomic write to ~/.config/styx/state/intel/<project>/index.json
```

### `intel ls`

Lists all cached indexes with freshness state (fresh / stale-commits / stale-days).

### `context show`

Prints the rendered `.claude/context.md` for the current project's intel index. Debug tool.

### `auto <goal>` (NEW — slice 3)

Full autonomous pipeline. See §14 for full mechanics. High-level shape:

```
styx auto "add streaming to /chat endpoint"

[styx] run id: 20260527-184502-add-streaming
[styx] stage 1/7 research — convergence loop running…
       round 1: 3 blocking, 5 important
       round 2: 0 blocking, 2 important
       round 3: CONVERGED
       brief: docs/research/20260527-add-streaming-brief.md
[styx] stage 2/7 intel — index fresh (HEAD abc1234, 0 commits since build) ✓
[styx] stage 3/7 plan — generating from brief + intel…
       plan: docs/plans/20260527-add-streaming-plan.md
[styx] stage 4/7 execute — applying plan on branch styx/20260527-add-streaming…
       claude is editing 6 files…
       commit abc5678: "feat(chat): add SSE streaming"
[styx] stage 5/7 test — running `make test`…
       all green (47 passed, 0 failed) ✓
[styx] stage 6/7 review — parallel multi-channel diff review…
       claude: 0 BLOCKING, 1 IMPORTANT
       codex:  0 BLOCKING, 0 IMPORTANT
       fix-loop: 1 attempt resolved IMPORTANT findings ✓
[styx] stage 7/7 ship — creating PR via gh…
       https://github.com/user/repo/pull/472

DONE in 14m32s. Resume any time with: styx auto --resume 20260527-184502-add-streaming
```

Flags:
- `--resume <run-id>` — continue an interrupted run from the last completed stage
- `--no-pr` — stop at commit (don't open a PR); useful when you want to push manually
- `--no-push` — stop at commit (don't push at all)

### `execute <plan-file>` (NEW — slice 3)

Non-interactive code execution from an existing plan markdown file. Same mechanics as `auto`'s execute stage, but standalone. Useful when you have a plan from elsewhere (e.g., `gsd-plan-phase` output) and want autonomous execution against it.

Calls `claude --dangerously-skip-permissions -p "implement this plan: <plan-content>"` with the project's `.claude/context.md` already in place. Runs tests after; if tests fail, enters the same fix-loop the auto pipeline uses (max 5 attempts).

### `runs ls` / `runs show <run-id>` (NEW — slice 3)

Lists all pipeline runs for the current project (and historical runs) and shows the detailed state of one run: per-stage start/end times, artifacts, failure logs, current stage if in-progress, status (`running` / `completed` / `paused` / `failed`).

### `deep-research` (removed)

Replaced by `research --deep`. A one-line shim survives for the old verb that prints `[styx] 'deep-research' is now 'research --deep' in v0.2` and forwards.

### Build process

`internal/intel/intel.go::Build(project)`:

1. **Walk repo** respecting `.gitignore` + a built-in `EXCLUDED_DIRS` list (`node_modules`, `__pycache__`, `.venv`, `target`, `dist`, `.next`, etc.). Capture file paths into `file_tree`.
2. **Sniff conventions** (`internal/intel/sniff.go`, all deterministic, no LLM):
   - Test framework: presence of `pytest`/`jest`/`vitest`/`cargo test` indicators in manifests/configs
   - Type system: tsconfig, mypy.ini/pyproject `[tool.mypy]`, `Cargo.toml` features
   - Naming: sample 20 source files, infer dominant case
   - Imports: grep for absolute vs relative import patterns
   - Error handling: grep for common patterns (`raise`, `panic`, `throw`, `Result<`)
3. **Recent commits:** `git log -n 20 --pretty=format:%H|%s|%n%-(files changed)`
4. **Open TODOs:** `git grep -nE 'TODO|FIXME|HACK'` (cap to first 30)
5. **External deps:** parse manifests, extract direct deps
6. **Module summaries (LLM):** for each top-level dir, dispatch `agy -p "<module-summary-prompt>" --add-dir <module-path>`. Run with `errgroup` + semaphore (default concurrency = 4). Each call ~5-15s.
7. **Key symbols (LLM):** single `agy -p "<key-symbols-prompt>" --add-dir <project-root>` call.
8. **Assemble + atomic write:** marshal JSON, write to `<dir>/index.json.tmp`, rename to `index.json`.

Approximate build cost: 6-12 agy calls per build for a typical project. ~$0.50-2 worth of Gemini Pro tokens.

### Freshness

`intel.IsStale(project)`:

```go
func IsStale(p Project) bool {
    idx, err := Load(p)
    if err != nil { return true } // never built
    if time.Since(idx.BuiltAt) > 7*24*time.Hour { return true }
    if commitsSince(p.Path, idx.GitHead) > 5 { return true }
    return false
}
```

When stale, `plan`/`build` print `[styx] intel index stale (N commits / D days since rebuild) — rebuilding…` and call `Build(project)` synchronously before proceeding.

### Rendering for Claude

`internal/intel/render.go::ToMarkdown(index, brief) string` — produces the markdown shown in §5 above, written to `<project>/.claude/context.md` (or `context.styx.md` if a user-authored one exists).

## 7. Research convergence loop

### Critique format

System prompt to codex:

```
You are reviewing a research draft. Return your findings as JSON:
{
  "blocking": ["finding 1", "finding 2"],
  "important": ["finding 1"],
  "nits": ["finding 1"]
}

BLOCKING:  factual errors, untested claims that change the conclusion, missing critical context.
IMPORTANT: weak evidence, alternative interpretations not addressed, edge cases that matter.
NIT:       style, wording, minor omissions.

Be specific. Return ONLY the JSON object, no surrounding text.
```

### Parser

```go
type Critique struct {
    Blocking  []string
    Important []string
    Nits      []string
}

func Parse(raw string) (Critique, error) {
    // 1. Try strict JSON unmarshal
    if c, err := tryJSON(raw); err == nil { return c, nil }

    // 2. Try extracting a JSON object from within prose (regex-bracketed match)
    if c, err := tryEmbeddedJSON(raw); err == nil { return c, nil }

    // 3. Fall back to keyword extraction (sections starting with "BLOCKING:", etc.)
    return parseKeywordSections(raw)
}
```

### Convergence + oscillation

```go
const maxRounds = 6

func Loop(query string, drafter, critic Channel) (*Brief, error) {
    var drafts []string
    drafts = append(drafts, drafter.Draft(query))

    for round := 1; round <= maxRounds; round++ {
        crit := critic.Critique(drafts[len(drafts)-1])

        if len(crit.Blocking) == 0 && len(crit.Important) == 0 {
            return &Brief{Drafts: drafts, Critiques: critiques, Status: "converged"}, nil
        }

        // Oscillation: if this draft would equal the one 2 rounds back
        if round >= 3 && hash(drafts[round-1]) == hash(drafts[round-3]) {
            return &Brief{..., Status: "oscillating"}, nil
        }

        drafts = append(drafts, drafter.Revise(drafts[len(drafts)-1], crit))
    }
    return &Brief{..., Status: "max_rounds_exhausted"}, nil
}
```

### `--deep` extension

After the loop returns (regardless of status), if `--deep`:

```
1. Extract URLs from the final draft (regex).
2. For each unique URL, parallel-fetch (HTTP GET, 30s timeout, 5 concurrent).
3. For each fetched body, dispatch agy summarize.
4. Append "Sources" section to the brief.
5. One final critic pass on the source-rich draft.
```

## 9. Error handling additions over v0.1

| Failure | Behavior |
|---|---|
| `agy` not installed (binary missing on PATH at startup) | Detected lazily on first agy-routed call. Print install command + exit code 2. Routing's fallback chain kicks in for downstream calls. |
| `agy` auth not configured | `agy` itself prints an auth URL on first call. Styx detects this in stderr, surfaces it to user, exits cleanly. |
| Intel build interrupted | Atomic write protects prior index; new build starts over (no resume in v0.2). |
| Convergence loop oscillating | Content-hash detection. Brief marked `Status: oscillating`, both drafts + critiques included. Not an error — gives the artifact. |
| Codex critic returns invalid JSON | Three-tier parser: strict JSON → embedded JSON regex → keyword extraction. If all three fail, treat response as a single IMPORTANT finding. |
| `.claude/context.md` already exists, user-authored | Write to `context.styx.md`, reference via `@import` line in canonical file. Never clobber. |
| Repo has > 50k files | Hard cap on `file_tree` (truncate with notice). Modules still summarized per-dir; only the file list is capped. |
| Agy returns non-JSON-shaped output to a "summarize this module" prompt | Treat the raw response as the `purpose` field. Module still gets recorded with degraded data. |

### Global error precedence

If `styx plan` fails any of: intel build, brief load, context write, claude call — emit a structured `[styx] step failed: <step>` message with the actual error, then exit code 1. Earlier steps' artifacts (e.g., refreshed intel index) are kept.

- `auto_review_fix_loop.txt` — review finds 2 BLOCKING; one attempt resolves; pipeline continues to ship
- `auto_resume_after_test_exhaustion.txt` — first run halts at test-stage exhaustion; resume after manual fix; pipeline completes
- `auto_no_pr_mode.txt` — `--no-pr` flag stops at push, no PR opened, status=completed
- `auto_concurrent_run_rejected.txt` — second `auto` invocation in same project hits the lock and exits cleanly

All use fake channels + fake `gh`/`git` shims (no network).

## 11. Migration v0.1 → v0.2

Automatic on first v0.2 startup:

1. Detect any `gemini:*` references in `~/.config/styx/routing.toml`.
2. If found, run `styx upgrade`:
   - Back up to `routing.v0.1.toml.bak`

## 12. Open questions / risks

1. **No `--model` flag on `agy`.** Gemini model selection appears to be server-side. If we need to force Flash vs Pro for cost reasons, we may need to extend the routing key (`agy:flash` vs `agy:pro`) and pass it as some other parameter (env var? plugin config?). v0.2 ships with a single `agy:default` and accepts whatever model the user's subscription serves.
2. **`agy` rate limits / quota visibility.** The `agy` CLI doesn't expose a usage endpoint. Budget tracking falls back to local request-count + estimated-token math. May undercount or overcount.
3. **Intel index module-summary parallelism.** Concurrent `agy` calls (semaphore = 4) may hit rate limits during builds of large repos. Plan: catch rate-limit errors, reduce concurrency, retry with backoff. Detail in implementation plan.
4. **Convergence loop runaway.** If critic perpetually finds new blocking items (adversarial prompt or genuine deep topic), 6 rounds × ~30-60s = 3-6 minutes per research call. The cap protects against infinite spend but may stop short of true convergence. Acceptable for v0.2; revisit if user reports issues.
5. **`.claude/context.md` collision.** Some users version-control this file. Styx's "write context.styx.md and reference it" approach avoids clobbering but creates a second file in the tree. Could be confusing. Plan: document clearly in README and on first write.
6. **Codex critique JSON adherence.** GPT-5 generally follows JSON instructions but may add prose. Three-tier parser handles this; if real-world data shows >5% parse-fallback rate, we'll add explicit JSON-mode flags to the codex CLI invocation.
7. **`claude --dangerously-skip-permissions` trust boundary.** The autonomous execute stage uses this flag, meaning Claude can edit any file, run any command, and the user has accepted that risk by invoking `styx auto`. Mitigations: branch is created per-run (so `main` is safe); pipeline state is preserved (so destructive actions can be inspected). A `--dry-run` simulation mode could be added in a later release if needed, but v0.2 ships without it per YAGNI.
8. **Fix-loops can run indefinitely on adversarial failures.** Test fix-loop is capped at 5 attempts, review at 3. After cap, halt + save state. The cost of a runaway is bounded (5×Claude calls per test attempt × $0.50 ≈ $2-3 ceiling). Acceptable.
9. **Pipeline state corruption.** If `state.json` is malformed (manual edit, partial write), resume fails. Atomic writes (tmpfile + rename) protect against partial writes during execution. Manual corruption is user's problem — we fail loudly rather than guess.
10. **`gh` not authenticated.** Pipeline ship stage falls back to "branch pushed, PR not opened" gracefully. User runs `gh auth login` once.
11. **The pipeline tests its own conventions detection.** If `intel`'s conventions sniffer mis-detects the test framework (e.g., misses a custom test runner), stage 5 won't know how to run tests and skips it. This is a silent failure mode; we mitigate by printing the detected test command before running it ("`[styx] running tests via: pytest`") so the user notices if it's wrong.

## 14. Pipeline: `styx auto <goal>`

### High-level flow

```
                  ┌────────────────────────────────────────────┐
                  │       styx auto "<goal>"                    │
                  │       run-id = <timestamp>-<slug>           │
                  └─────────────────────┬───────────────────────┘
                                        │
   ┌────────────────────────────────────┼────────────────────────────────────┐
   │                                    │                                     │
   ▼                                    ▼                                     ▼
[1] research       [2] intel        [3] plan       [4] execute      [5] test
convergence loop   refresh if       claude -p      claude -p with   make test
agy + codex        stale            with brief +   --dangerously-   (or sniffed
6 rounds max       (auto)           intel ctx      skip-perms       test cmd)
   │                                                   │                ▲
   │                                                   │                │
   │                                                   ▼                │ fix-loop
   │                                              edits applied         │ max 5
   │                                              commits made          │
   │                                                   │                │
   │                                                   └────────────────┘
   │
   ▼                                            ▼
[6] review                              [7] ship
parallel claude + codex                 gh pr create
findings filtered to BLOCKING/IMPORTANT  branch pushed
   │                                            │
   │ fix-loop max 3                             │
   ▼                                            ▼
review pass                             DONE
```

### Stages

#### Stage 1: Research

- Skip if a recent brief exists for a closely-matching query (fuzzy match on slug or recent N briefs include the goal text). Configurable via `--force-research`.
- Otherwise: invokes the convergence loop from §7 (deep, citation chasing OFF by default — `--deep` flag on `auto` enables it).
- Failure: if loop returns `oscillating` or `max_rounds_exhausted` after 6 rounds, **continue** with the best-effort brief. Mark in run state.
- Output: brief markdown file path.

#### Stage 2: Intel refresh

- Standard freshness check (§6). If stale, rebuild before proceeding.
- Output: intel index path (always present after this stage).

#### Stage 3: Plan

- Use latest brief (from stage 1 if new, else most recent).
- `.claude/context.md` already materialized by stage 2.
- Calls `claude -p "<plan prompt with goal + brief + intel inline>"` (one-shot, no tool use here — plan is a markdown writeup).
- Output: plan markdown path.

#### Stage 4: Execute

- Creates a feature branch `styx/<run-id>` from current HEAD if not already on one (per decision #13).
- Calls `claude --dangerously-skip-permissions -p "implement this plan: <plan-content>"`. Claude Code's harness handles file edits, running commands, etc. The `.claude/context.md` is auto-loaded.
- After Claude returns, styx checks `git status`:
  - If there are uncommitted changes → styx stages all and commits with `feat(<scope>): <goal-slug>` (derived from the goal).
  - If there are no changes → stage fails with "Claude produced no edits."

#### Stage 5: Test

- Sniff the test command from the intel index's `conventions.test_framework`:
  - `pytest` → run `pytest`
  - `jest`/`vitest` → run `npm test` (or detected `package.json` script)
  - `go` → run `go test ./...`
  - `cargo` → run `cargo test`
  - else: skip with `[styx] no test framework detected; skipping test stage` (still proceeds to review).
- If tests pass → next stage.
- If tests fail → **fix-loop** (max 5 attempts):
  - Send Claude (`--dangerously-skip-permissions -p`) the test output + the original plan + the diff so far. Prompt: "Tests failed. Diagnose and fix. Commit fixes as separate commits."
  - Re-run tests.
  - After 5 failed attempts, halt the pipeline. State is saved; user can `styx auto --resume` after intervention.

#### Stage 6: Review

- Standard `review` verb behavior from v0.1 (parallel Claude + Codex diff review with synthesis).
- Findings filtered to BLOCKING + IMPORTANT only (NITs ignored for autonomous flow).
- If 0 BLOCKING + 0 IMPORTANT → next stage.
- Otherwise → **fix-loop** (max 3 attempts):
  - Send Claude the findings + diff. "Fix every BLOCKING and IMPORTANT finding. Commit fixes."
  - Re-review.
  - After 3 attempts, halt + state saved.

#### Stage 7: Ship

- `git push -u origin styx/<run-id>` (skipped if `--no-push`).
- `gh pr create --fill --body "$(cat <(echo "Goal: <goal>"; echo; echo "Convergence brief: <brief-path>"; echo; echo "Plan: <plan-path>"; echo; echo "Generated by styx auto, run <run-id>"))"`.
  - Skipped if `--no-pr` flag set.
  - If `gh` not installed or no remote → degrade to "branch + commit done locally; PR not opened."
- Print success URL (or local-only notice).

### Pipeline state

Per-run dir: `<project>/.styx/runs/<run-id>/`

```
.styx/
└── runs/
    └── 20260527-184502-add-streaming/
        ├── state.json          # current stage + per-stage status
        ├── stage-1-research/
        │   └── brief.md        # symlink or copy of the brief
        ├── stage-3-plan/
        │   └── plan.md
        ├── stage-4-execute/
        │   ├── commits.txt     # list of commit SHAs from this stage
        │   └── claude-stdout.log
        ├── stage-5-test/
        │   ├── attempt-1-output.log
        │   └── attempt-2-output.log
        ├── stage-6-review/
        │   └── findings.json
        └── stage-7-ship/
            └── pr-url.txt
```

`.styx/` added to project's `.gitignore` (styx writes/appends this on first run).

`state.json` shape:

```jsonc
{
  "run_id": "20260527-184502-add-streaming",
  "goal": "add streaming to /chat endpoint",
  "started_at": "2026-05-27T18:45:02Z",
  "updated_at": "2026-05-27T18:59:34Z",
```

### Resume

`styx auto --resume <run-id>`:

1. Load `state.json`.
2. Verify branch still exists; checkout.
3. Pick up at `current_stage` (re-run if it was in-progress; skip if completed).
4. Continue forward.

If state is corrupt or branch deleted → fail with clear message; user can manually clean up.

### Concurrency / locking

- `styx auto` writes `<project>/.styx/runs/.lock` containing the current run-id at start.
- If another `styx auto` starts in the same project and the lock is held → reject: "Another pipeline is in progress: <run-id>. Resume it or remove `.styx/runs/.lock` if stale."
- Lock cleared on successful completion or explicit `styx auto --abort <run-id>`.

### Cost / time expectations

For a Hoot-sized backend feature (e.g., "add streaming to /chat"):

| Stage | Time | LLM cost |
|---|---|---|
| Research (3 rounds avg) | 2-4 min | $0.30-0.80 |
| Intel (cached) | 0-5s if fresh; 1-2 min if rebuild | $0-1 |
| Plan | 30s-1 min | $0.20 (Claude Sonnet) |
| Execute | 2-8 min | $0.50-2 (Claude with tools, multi-step) |
| Test + fix (avg 1-2 attempts) | 1-5 min | $0.30-1 |
| Review + fix (avg 1 attempt) | 30s-2 min | $0.50 |
| Ship | 5-10s | $0 |
| **Total** | **6-20 min** | **$2-5** |

The user is on a $100/mo Claude plan + $20/mo ChatGPT + free Gemini API. At $2-5 per auto-run, the Claude budget supports ~20-50 auto runs per month before degradation kicks in.

### `--no-pr` / `--no-push` modes

- `--no-pr`: stop at stage 7 with branch pushed but no PR. User opens PR manually.
- `--no-push`: stop at stage 7 with branch committed locally, no push. User pushes manually.
- Both: do the inverse stage 7 work; just don't reach the failure points.

### Halt and resume scenarios

| Trigger | Behavior |
|---|---|
| User Ctrl-C | Mark current stage as `paused`. State preserved. `styx auto --resume` picks up. |
| Test fix-loop exhausted | Mark `status: failed`, `failures: [...]`. User inspects, fixes, then `styx auto --resume <id>` re-enters from stage 5. |
| Review fix-loop exhausted | Same as test exhaustion but for stage 6. |
| Claude returns no edits | `execute` stage fails. Plan likely needs revision. User re-runs `styx plan` and then `styx auto --resume`. |
| Network failure mid-call | Channel-level retry (existing v0.1 logic). If retries exhaust, stage fails; resume after connectivity restored. |
| Branch already exists from prior run | Reject: "Branch styx/<run-id> already exists. Use `--resume <run-id>` or delete the branch." |

### Verbs the pipeline composes

The pipeline doesn't reimplement anything — every stage calls into existing styx code:

- Stage 1: `research.Loop()` (from §7)
- Stage 2: `intel.Build()` (from §6)
- Stage 3: `cmdPlan` (the standalone plan verb, called as a Go function)
- Stage 4: `execute.Apply(plan, project, dryRun=false)` (new `internal/execute/` package)
- Stage 5: `execute.RunTests()` + `execute.FixLoop()`
- Stage 6: `review.Run()` (extracted from v0.1 cmdReview) + `execute.FixLoop()`
- Stage 7: `git`/`gh` shell-outs

Because each stage is also exposed as a standalone verb, the user can always run them individually (decision #11). `auto` is composition, not magic.
